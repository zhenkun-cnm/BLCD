This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

STM32CubeMX-generated BLDC motor control project targeting **STM32F051K8Ux** (Cortex-M0, 64KB flash, 8KB SRAM). Built with **Keil MDK-ARM v5** (uVision) using **ARMCLANG v6.9**.

The project is in its early skeleton stage — no BLDC-specific peripherals or control logic have been added yet.

## Core Development Rules

These rules are mandatory and apply to all code written in this project:

### 1. Core Task
Sensorless BLDC/PMSM (brushless DC / permanent magnet synchronous motor) control development.

### 2. Pure Register-Level Development
- **Strictly forbidden**: HAL library, standard peripheral library, or any high-level abstraction libraries for peripheral access.
- All peripheral operations must directly access hardware registers.
- **Register bit manipulation** must use CMSIS-defined `_Pos` macros for readability — never use raw shift counts:
  ```c
  // Correct
  ADC1->CFGR1 |= (1u << ADC_CFGR1_OVRMOD_Pos);

  // Wrong — raw magic numbers are forbidden
  ADC1->CFGR1 |= (1u << 12);
  ```
- Add concise comments on non-obvious register operations.

### 3. High Portability Architecture
- **Algorithm layer** (motor control math, FOC, observer) and **hardware driver layer** (PWM, ADC, GPIO) must be strictly decoupled.
- Hardware accesses must be wrapped through **macro definitions** or **interface functions** so the algorithm code can be ported to a different MCU by rewriting only the driver layer.
- Example pattern:
  ```c
  // In hardware driver header (MCU-specific)
  #define BLDCDRV_PWM_SET_DUTY(ch, val)  (TIM1->CCR##ch = (val))

  // In algorithm code (MCU-agnostic)
  BLDCDRV_PWM_SET_DUTY(1, duty_a);
  BLDCDRV_PWM_SET_DUTY(2, duty_b);
  ```

## Build System

- **IDE**: Keil uVision 5 (`.uvprojx` project file at `MDK-ARM/BLCD.uvprojx`)
- **Toolchain**: ARMCLANG v6.9 (AC6)
- **No CLI build** — compilation and flashing happen through the Keil IDE
- Output artifacts: `BLCD.axf` (ELF), `BLCD.hex` (Intel Hex) in `MDK-ARM/BLCD/`
- Debug probe: ST-Link (via STLink SWO DLL)
- Flash driver: STM32F0xx 64KB Flash

## Memory Layout

From the scatter file (`MDK-ARM/BLCD/BLCD_sct.Bak`):

| Region | Start       | Size      | Usage      |
| ------ | ----------- | --------- | ---------- |
| Flash  | `0x08000000` | `0x10000` | Code + RO  |
| SRAM   | `0x20000000` | `0x02000` | RW + ZI data |

Stack: `0x400` bytes, Heap: `0x200` bytes (defined in `startup_stm32f051x8.s`). 8KB SRAM is tight — be mindful of stack depth and global allocations.

## Architecture

### Current codebase (skeleton)

- **main.c** — Application entry. `main()` runs the infinite loop; currently empty.
- **stm32f0xx_it.c** — Interrupt handlers. `SysTick_Handler` calls `HAL_IncTick()` for HAL timing. SVC, PendSV, HardFault, and NMI handlers are present (with empty or infinite-loop defaults). All other IRQ handlers default to weak aliases from the startup file.
- **gpio.c / MX_GPIO_Init()** — GPIO initialization. Currently only enables GPIOA clock (PA13/PA14 for SWD).
- **stm32f0xx_hal_msp.c / HAL_MspInit()** — MCU Support Package init. Enables SYSCFG and PWR clocks.
- **system_stm32f0xx.c / SystemInit()** — Clock configuration called before `main()`. Configures PLL for 48MHz system clock from 8MHz HSI.

### Target architecture (going forward)

The project will move to a **two-layer architecture**:

1. **Hardware Driver Layer** (`Drv/` or similar) — MCU-specific register-level code
   - PWM timer control, ADC sampling, GPIO toggling
   - All register accesses live here, wrapped as macros or inline functions
   - This is the only layer that touches hardware registers directly

2. **Algorithm Layer** (`Algo/` or similar) — MCU-agnostic motor control
   - FOC algorithm, observer/sensorless estimation, control loops
   - Calls driver layer through abstracted interfaces only
   - No direct register access, no CMSIS includes

## Known Gotcha: TIM Compilation

The TIM HAL source files (`stm32f0xx_hal_tim.c`, `stm32f0xx_hal_tim_ex.c`) are compiled into the project, but `HAL_TIM_MODULE_ENABLED` is **commented out** in `stm32f0xx_hal_conf.h`. This means the TIM HAL functions compile to near-empty objects (guarded by `#ifdef`). If you enable a TIM peripheral in CubeMX, also uncomment `#define HAL_TIM_MODULE_ENABLED` in the HAL config — otherwise timer code will silently do nothing. This is only relevant if HAL TIM is used; under register-level development, TIM registers are accessed directly.
